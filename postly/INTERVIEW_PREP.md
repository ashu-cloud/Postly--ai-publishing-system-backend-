## Technical Decision Deep-Dives

### 1. Why Prisma over Knex?

**Type safety at every layer.** Knex is a query builder — you write SQL-like chains but TypeScript can't infer what the result shape will be without manual type annotations. Prisma generates a fully typed client from your schema, so `prisma.user.findUnique({ where: { email } })` returns `User | null` — the exact correct type, automatically.

**Schema as source of truth.** With Prisma, `schema.prisma` defines your data model. Migrations are auto-generated by diffing the current DB state against the schema. With Knex, you write migrations manually — schema drift (DB vs code) is a common source of bugs. Prisma makes it structurally impossible for your models to drift from your migrations.

**Developer ergonomics.** Prisma Studio gives a visual DB explorer. `npx prisma migrate dev` handles dev migrations. The generated client has autocomplete on every field and relation.

**Trade-off:** Prisma is slightly less flexible for very complex raw SQL (though you can use `prisma.$queryRaw`). For this project, the type safety benefit outweighs the flexibility loss.

---

### 2. Why grammy over node-telegram-bot-api?

**TypeScript-native.** `node-telegram-bot-api` was written before TypeScript was mainstream — its types are added as a community package and often lag behind the actual API. `grammy` was built from the ground up in TypeScript. Every context property, every callback, every event is fully typed.

**Middleware model.** grammy uses a Koa-style middleware chain: `bot.use(middleware)`. This is composable and testable. `node-telegram-bot-api` uses event emitters — harder to intercept, modify, and test.

**Webhook handling.** grammy's `webhookCallback(bot, 'express')` returns a standard Express middleware. Switching between webhook and polling is one line change (`bot.start()` vs `webhookCallback`). `node-telegram-bot-api` has a more complex webhook setup with its own HTTP server.

---

### 3. Why AES-256-GCM over AES-256-CBC?

**Authenticated encryption.** GCM (Galois/Counter Mode) provides two things: **confidentiality** (ciphertext can't be read) and **integrity** (ciphertext can't be tampered with undetected). CBC provides only confidentiality.

**Padding oracle attacks.** CBC uses padding (PKCS7) to align data to block boundaries. By making many decryption attempts with crafted ciphertexts and observing error responses, an attacker can decrypt the data one byte at a time. This is the padding oracle attack. GCM has no padding — this entire attack class doesn't apply.

**Malleability.** CBC ciphertext is malleable — an attacker can flip specific bits in the ciphertext to predictably flip bits in the decrypted plaintext. GCM's authentication tag detects any modification.

**The encryption format:** `iv:authTag:ciphertext`
- `iv`: 16 random bytes per encryption — identical plaintexts produce different ciphertexts (semantic security)
- `authTag`: 16-byte GCM tag — any modification to the ciphertext causes decryption to throw
- `ciphertext`: the encrypted payload

---

### 4. Why bcrypt cost factor 12?

**OWASP recommendation.** OWASP's Password Storage Cheat Sheet recommends bcrypt with a minimum cost factor of 10, and recommends 12 as a good modern default.

**GPU resistance.** bcrypt's design is intentionally sequential — it can't be parallelised across GPU cores the way MD5 or SHA can. At cost 12, a modern GPU can attempt ~1000 hashes/second vs ~10^9/second for SHA-256.

**Acceptable latency.** Cost 12 takes ~250ms on modern hardware. For login, this is imperceptible to users (they're waiting for a page load anyway). For an attacker brute-forcing 10^9 passwords, 250ms × 10^9 = ~8000 years.

**Future-proofing.** bcrypt allows increasing the cost factor. When you increase it, you can re-hash users' passwords on their next successful login (transparent to the user). You don't need to invalidate all passwords.

### 5. Why refresh tokens are stored in DB?

**Stateless JWTs cannot be invalidated.** A JWT's validity is determined only by its signature and expiry. If a user's device is stolen, you can't invalidate their access token before it expires. Short expiry (15 minutes) limits damage, but 15 minutes is still a long time.

**Refresh tokens need revocation.** If a refresh token is stolen, an attacker can use it to get new access tokens indefinitely. By storing refresh tokens in PostgreSQL with a `revoked` boolean, we can:
- **Logout**: mark token as revoked immediately
- **Forced logout**: delete all tokens for a user (admin action, security incident)
- **Token rotation**: revoke old token on every refresh call — stolen tokens become one-time-use

**Token rotation as theft detection.** When a legitimate user refreshes, the old token is revoked and a new one is issued. If an attacker later uses the stolen old token, it's already revoked — and the legitimate user's next refresh will fail (the new token was also stolen). This mutual failure signals theft.

### 6. Why one BullMQ job per platform per post?

**Independent retry.** If Twitter fails 3 times, we don't want to discard the LinkedIn job. Each platform job retries independently — Twitter can be retried later without touching LinkedIn.

**Isolated failure domains.** A LinkedIn API outage causes LinkedIn jobs to back up in the LinkedIn queue. Twitter, Instagram, and Threads queues are unaffected.

**Platform-specific rate limiting.** Twitter allows 300 posts per 15 minutes. LinkedIn allows 150. By having separate queues, we can configure different concurrency levels per queue to respect each platform's rate limits.

**Granular status tracking.** `PlatformPost` records each have their own `status`, `attempts`, `errorMessage`, and `publishedAt`. This is only possible if each platform is an independent job.

---

### 7. Why exponential backoff?

**Thundering herd prevention.** If a platform API goes down and all jobs retry simultaneously, the sudden flood of requests can prevent the API from recovering. Exponential backoff spreads retries over time.

**Transient failure recovery.** Most platform API failures are transient (brief outages, rate limit windows). Waiting immediate → 5s → 25s gives the platform time to recover before the next attempt.

**Implementation detail:** We use a named BullMQ backoff strategy that maps retries to:
- Attempt 1: immediate
- Attempt 2: 5000ms
- Attempt 3: 25000ms

---

### 8. Why Redis for bot session state?

**TTL support.** Redis has native key expiry. We set `EXPIRE bot:session:{chatId} 1800` on every state update — the session disappears automatically after 30 minutes of inactivity. This is impossible with in-process Maps.

**In-memory speed.** Redis is sub-millisecond for get/set operations. Bot conversations require fast state access on every message — DB round trips would be too slow.

**Multi-instance safe.** In production, the app might run on multiple instances (horizontal scaling). An in-process Map is local to one process — session state would be lost if the user's request hits a different instance. Redis is shared across all instances.

**Survives restarts.** If the server restarts (deploy, crash), Redis still has the session data. An in-process Map would be wiped.

---

### 9. Why `cuid()` over `uuid()`?

**Shorter.** cuid is ~25 characters. UUID is 36 characters (with hyphens). Shorter IDs = smaller indexes, shorter URLs, smaller payloads.

**URL-safe.** cuid uses only `[a-z0-9]` — no hyphens or special characters. UUIDs have hyphens that need percent-encoding in URLs.

**K-sortable.** cuids are monotonically increasing — newer records sort after older ones. This means INSERT operations on a primary key index are always appending to the end (no page splits in B-tree indexes). UUIDs are random — each INSERT can go anywhere in the B-tree, causing frequent page splits and index fragmentation over time. For a write-heavy table like `PlatformPost`, this matters.

---

### 10. Why separate `platform_posts` table?

**One post → many outcomes.** A single publish request (one `Post`) produces one `PlatformPost` per platform. Each has an independent: `status`, `errorMessage`, `attempts`, `publishedAt`, `content`.

**Independent status tracking.** Twitter can be PUBLISHED while LinkedIn is FAILED. You can't represent two different statuses in a single `posts` row without storing them as JSON (losing the ability to index and query efficiently).

**Retry granularity.** The retry endpoint re-queues only FAILED `PlatformPost` records for a given `Post`. This is only possible if platforms are separate records.

**Dashboard queries.** "How many posts did I publish to Twitter this month?" requires `GROUP BY platform` on `platform_posts`, not `posts`.

### 11. Token Rotation Flow (Complete)

1. Client has: `access_token` (15min) + `refresh_token` (UUID in DB, 7 days)
2. Access token expires → client calls `POST /api/auth/refresh` with `refresh_token`
3. Server looks up `refresh_token` in `RefreshToken` table
4. Checks: exists? not revoked? not past `expiresAt`?
5. If valid: `UPDATE refresh_tokens SET revoked=true WHERE id=...` (old token killed)
6. Creates new refresh token in DB (new UUID, new 7-day expiry)
7. Generates new access token (JWT, 15min)
8. Returns both tokens
9. If attacker uses old token: step 4 fails (revoked=true) → 401

**Replay attack prevention:** Each refresh token can only be used once. On use, it's immediately revoked and a new one is issued. An intercepted token is only valid until the legitimate client uses it — at which point the stolen token becomes worthless.

### 12. Partial Failure Handling

```
User publishes to [TWITTER, LINKEDIN]
  → Creates Post { status: QUEUED }
  → Creates PlatformPost(TWITTER) { status: QUEUED }
  → Creates PlatformPost(LINKEDIN) { status: QUEUED }
  → Queues 2 BullMQ jobs

Twitter job runs:
  → PlatformPost(TWITTER).status = PROCESSING
  → Calls twitter.service.ts
  → Twitter API returns 429 Rate Limited
  → Throws error → BullMQ retries 3x
  → Final failure: PlatformPost(TWITTER).status = FAILED
  → recomputePostStatus() → mix of FAILED+QUEUED → Post.status = PROCESSING

LinkedIn job runs:
  → PlatformPost(LINKEDIN).status = PROCESSING
  → Calls linkedin.service.ts
  → LinkedIn API returns 200
  → PlatformPost(LINKEDIN).status = PUBLISHED
  → recomputePostStatus() → PUBLISHED + FAILED = PARTIAL → Post.status = PARTIAL

User calls POST /api/posts/{id}/retry:
  → Finds FAILED platform posts: [PlatformPost(TWITTER)]
  → Resets attempts=0, status=QUEUED
  → Queues new Twitter job
  → LinkedIn is NOT re-queued (already PUBLISHED)
```

### 13. Prompt Engineering Approach

**System vs User prompt split:**
- **System prompt** → platform constraints + response format. Sent as `role: "system"`. Models treat this as higher-priority instructions than user content.
- **User prompt** → the actual content request (post type, tone, idea). Sent as `role: "user"`.

**Why constraints in system prompt:** If you put character limits in the user message, they can be "forgotten" by the model when it's focused on generating creative content. System prompts are more persistent across the generation process.

**Tone injection:** Tone is injected in the user prompt (`"Create content with a {tone} tone"`) because tone is content guidance, not structural constraint.

**JSON format enforcement:** The JSON response format is placed at the very END of the system prompt — recency bias means the model processes it last and it's freshest in context when generating output.

**Fallback parsing:** The response parser strips markdown code fences (` ```json `) because models frequently add them despite "return ONLY JSON" instructions. Defensive parsing > fighting the model.

### 14. Encryption Format Explained

Format stored in DB: `iv:authTag:ciphertext`

Example: `a1b2c3d4e5f60708:deadbeefcafeface...:4f3e2a1b...`

**Why separate sections?**
- `iv`: Must be stored alongside ciphertext — needed for decryption. By embedding it in the same field, we avoid a separate DB column.
- `authTag`: Must be stored alongside ciphertext — needed to verify integrity on decryption. Same rationale.
- Colon separator: Unambiguous delimiter. Hex strings never contain colons, so splitting on `:` always gives exactly 3 parts.

**Why hex encoding?** Binary data would require binary DB columns or base64. Hex is simpler, portable, and readable in logs.

**Why random IV per encryption?** If the same IV is used with the same key for two plaintexts, an attacker who knows one plaintext can XOR it with the ciphertext to recover the keystream and decrypt other plaintexts encrypted with the same IV+key. Random IV ensures this is impossible even if two users store the same token value.

## Interview Questions — Complete Answers

---

### "Walk me through what happens when a user sends /post in the Telegram bot"

"When the user sends /post, Telegram delivers an update to our POST /api/bot/webhook endpoint. grammy dispatches it to the bot.command('post') handler, which calls startPublishFlow(). This creates a BotSession object in Redis at key `bot:session:{chatId}` with `step: 'POST_TYPE'` and 1800-second TTL. It then sends a message with an inline keyboard showing post type options.

When the user taps a button, Telegram sends a callback_query update to the webhook. The handleCallbackQuery() function reads the session from Redis, sees that the step is POST_TYPE, extracts the selected value from the callback data (e.g., `pt:ANNOUNCEMENT`), updates the session to `step: 'PLATFORMS'`, saves it back to Redis (resetting the TTL), and sends the platform selection keyboard.

This continues through TONE and MODEL steps. On the IDEA step, we wait for free text — the `bot.on('message:text')` handler fires, checks the session, and if step is IDEA, calls handlePublishConversation(). The idea is validated (max 500 chars), then we call the AI generation service which returns platform-specific content. The content is stored in the session as generatedContent, step moves to CONFIRM, and the formatted preview is sent to the user.

When the user taps 'Post Now', we call publishPost() with the session data — the same service used by the REST API — which generates the content (or uses the cached generated content), creates DB records, and queues BullMQ jobs. The session is then cleared from Redis."

### "How does your refresh token rotation prevent replay attacks?"

"The key insight is that each refresh token can only be used once. When a client calls /auth/refresh, we:
1. Look up the token in the DB
2. Mark it as revoked immediately
3. Issue a new refresh token

If an attacker intercepts a refresh token and tries to use it, two scenarios:
- If the attacker uses it before the legitimate client: the legitimate client's next refresh call will find their token revoked and get a 401, alerting them that something is wrong.
- If the legitimate client uses it first: the attacker tries to use the now-revoked token and gets a 401 — the stolen token is worthless.

In both cases, the stolen token has a maximum useful lifetime of one refresh cycle. This is fundamentally more secure than JWTs where a stolen refresh token remains valid for its full 7-day window with no way to revoke it."

---

### "If Twitter fails but LinkedIn succeeds, what does your system do?"

"Each platform is an independent BullMQ job with independent status tracking in the PlatformPost table.

When LinkedIn succeeds, its worker updates PlatformPost(LINKEDIN).status to PUBLISHED and calls recomputePostStatus(). At this point, Twitter is still FAILED, LinkedIn is PUBLISHED — the function sees a mix, so it sets Post.status = PARTIAL.

Twitter has failed after 3 attempts: PlatformPost(TWITTER).status = FAILED, errorMessage recorded.

The user can call POST /api/posts/{id}/retry. The service finds all FAILED PlatformPost records for this Post — just Twitter. It resets Twitter's attempts to 0 and status to QUEUED, then queues a new BullMQ job for Twitter only. LinkedIn is not re-queued because it's already PUBLISHED. This prevents double-posting."

---

### "Why did you choose BullMQ over a simple setTimeout for scheduling?"

"setTimeout is process-local and ephemeral — if the server restarts, all pending timeouts are lost. A SCHEDULED post would never get published.

BullMQ stores jobs in Redis. If the server restarts, jobs are still in Redis. When the server comes back up, workers pick up where they left off. BullMQ also provides:
- Retry logic with exponential backoff (built-in)
- Job status visibility (dashboard can query BullMQ job states)
- Concurrency control (limit simultaneous API calls per platform)
- Delayed jobs (publishAt - now = delay in ms, stored in Redis sorted set)
- Dead letter queues (failed jobs aren't lost)

setTimeout is fine for very short-lived delays where data loss is acceptable. For a production publishing system, data durability is non-negotiable."

---

### "How would you scale this to 10,000 concurrent users?"

"Current bottlenecks and solutions:

**Database:** Add read replicas for dashboard/list queries. Use connection pooling (PgBouncer) — Prisma opens one connection per worker, which can exhaust PostgreSQL's connection limit at scale.

**Redis:** Use Redis Cluster — data is sharded across multiple nodes. BullMQ supports cluster mode.

**Workers:** BullMQ workers are independently scalable. We can run 10 Twitter workers on a dedicated machine while keeping 2 LinkedIn workers. Add workers until queue depth stabilizes.

**API server:** Stateless Express app — scale horizontally with a load balancer. Session state is in Redis (not in-process), so any instance can handle any request.

**AI generation:** This is the main bottleneck — each generate call takes 1-5 seconds. Solutions: request queuing, user-level rate limiting on /api/content/generate, caching for identical idea+settings combinations.

**Telegram bot:** grammy supports webhook mode with load balancing. Each webhook request is independent — no shared state between requests except Redis."

---

### "What's a race condition that could happen in your queue and how would you prevent it?"

"Race condition: Two workers both process the same PlatformPost record simultaneously. This could happen if a job is delayed and BullMQ's lock expires before the worker finishes (BullMQ uses SETNX locks in Redis).

The first worker would update PlatformPost to PUBLISHED. The second worker would try the same update and also succeed. Worse, it would call the platform API twice — resulting in a duplicate post.

Prevention strategies:
1. **BullMQ lock timeout:** Set a generous lock timeout (`lockDuration: 30000` ms) so the job lock doesn't expire during normal processing.
2. **Idempotency check:** Before calling the platform API, check if PlatformPost.status is already PUBLISHED. If so, skip the API call and return success.
3. **Platform API idempotency keys:** Some APIs (Twitter, LinkedIn) support idempotency keys in request headers. Include the platformPostId as the idempotency key — the platform will deduplicate the request.

In the current implementation, the status check is the first line of defence."

---

### "Why is AES-256-GCM better than AES-256-CBC?"

"Three reasons: authentication, malleability, and padding.

CBC doesn't authenticate the ciphertext — it only encrypts. An attacker who can submit modified ciphertexts to a decryption oracle can gradually recover the plaintext one byte at a time through padding oracle attacks. This is a well-known vulnerability in TLS 1.0 (BEAST, POODLE).

GCM computes an authentication tag over the ciphertext. Any modification — even one bit flip — causes decryption to throw an error. The attacker can't modify the ciphertext without detection, so oracle attacks are impossible.

GCM is also non-malleable — you can't predict how modifying the ciphertext affects the plaintext. CBC is malleable — specific ciphertext modifications cause predictable plaintext modifications.

Finally, GCM has no padding (it uses CTR mode internally). No padding = no padding oracle attack surface."

### "How does your rate limiting work?"

"Redis-based sliding window counter, keyed by user ID (or IP for unauthenticated requests). On every request:

1. `INCR rate:{userId}` — atomic increment
2. If the result is 1 (first request in window), `EXPIRE rate:{userId} 900` (15 minutes)
3. If current count > 100 → return 429

The `INCR + EXPIRE` approach is a counter window (not a true sliding window — the count resets every 15 minutes from the first request in that window, not a rolling window). A true sliding window would use a Redis sorted set with timestamps and `ZREMRANGEBYSCORE`.

The current approach is simpler and good enough for rate limiting. The remaining count is sent in `X-RateLimit-Remaining` response headers.

If Redis is unavailable, we fail open (allow the request) — it's better to let a few extra requests through than to take down the API during a Redis hiccup."

### "What indexes did you add and why?"

"I indexed based on the actual query patterns in the codebase:

`users.email` — every login call does `findUnique({ where: { email } })`. Without an index, this is a full table scan.

`refresh_tokens.token` — every `/auth/refresh` call does `findUnique({ where: { token } })`. Token lookups must be O(1).

`posts.userId` — every post query starts with `WHERE userId = ?`. The most common filter.

`posts.status` — dashboard queries filter by status (`WHERE userId=? AND status=?`). Without index, filters entire user's posts.

`posts.publishAt` — the scheduler polls `WHERE status=SCHEDULED AND publishAt <= NOW()` every 60 seconds. Without index, full table scan every minute.

`platform_posts.postId` — every `post.findUnique(...include platformPosts)` does a join on this key.

`platform_posts.status` — dashboard counts and worker job selection filter by status.

`platform_posts.platform` — dashboard: posts per platform requires `GROUP BY platform`."

### "What would you do differently if you had more time?"

"Several things:

**Full OAuth flows:** Currently tokens are manually provided. I'd implement the full OAuth 2.0 callback flow for Twitter and LinkedIn — the redirect to platform login, callback handling, token exchange, and refresh.

**WebSocket notifications:** Currently users check status via /status. I'd add a WebSocket connection so the bot can push 'Your LinkedIn post was published!' as soon as it happens, without the user asking.

**Idempotency keys:** Add idempotency key support on /api/posts/publish to prevent duplicate posts if a client retries after a network timeout.

**Queue monitoring:** Add a /api/admin/queues endpoint showing real-time queue depths, job counts by status, and worker health.

**Test coverage:** Current tests focus on validation and middleware. I'd add proper DB-seeded integration tests for the full publish flow using test transactions that roll back after each test.

**Content caching:** Cache AI generation results for 1 hour for identical (idea + platform + tone + model) combinations — reduces API costs significantly.

**Webhook auto-registration:** Auto-register the Telegram webhook on startup in production, using the WEBHOOK_URL env var."
